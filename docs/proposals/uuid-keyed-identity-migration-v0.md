# UUID-keyed identity migration — v0 (proposal)

**Status:** proposal / design only. No code, no migration SQL, no commitments. Operator is the merge + sequencing gate.
**Date:** 2026-06-14
**Relates to:** PR #700 (handle uniqueness), `docs/ontology/identity.md`, `docs/ontology/harness-substrate-plurality.md`, `docs/ontology/plan.md` (A2/S4, S5, S1), BEAM Wave 3 (`docs/proposals/beam-wave-3-handler-dispatch.md`).

## 1. Problem: the schema contradicts the identity ontology

The ontology already declares a three-tier identity model with the **UUID as the sole identity key**:

- `docs/ontology/harness-substrate-plurality.md:212` — "`agent_id` as a public/structured handle, `display_name` as social/cosmetic" (UUID = identity key).
- The live onboard response encodes the same: `identity_is: "uuid"`, `registry.is_identity_key: true` for the UUID; `agent_id_is: "public_structured_handle"`, `public_handle.is_identity_key: false` for the structured handle.

The **storage layer never adopted this.** The structured handle is the primary key, and lineage hangs off it:

```sql
-- db/postgres/schema.sql
core.agents.id              TEXT PRIMARY KEY            -- the structured handle, e.g. mcp_20260428
core.agents.parent_agent_id TEXT REFERENCES core.agents(id)   -- lineage = self-FK on the handle
core.identities.agent_id            TEXT NOT NULL UNIQUE REFERENCES core.agents(id)
core.identities.parent_agent_id     TEXT NULL REFERENCES core.agents(id)
```

The handle is generated by `generate_structured_id` as `{interface}_{model}_{YYYYMMDD}` (`src/mcp_handlers/support/naming_helpers.py`). It is **low-cardinality by construction** — every model-less MCP agent on a given day collapses onto `mcp_{date}`; the residents *are* this namespace (Lumen = `mcp_20260428`, Sentinel = `mcp_20260407`, …).

### Consequence: "the ontology forks into mcp ids"

Because lineage is keyed on the handle:

- `onboard()` takes `parent_agent_id` **verbatim** (`src/mcp_handlers/identity/handlers.py:1646`, commented "UUID of predecessor" but never resolved) and writes it into the AGE graph via `_create_spawned_edge_bg` → `create_agent_node(parent_id)` → `MERGE (a:Agent {id: $id})` (`src/db/age_queries.py:112`). The lineage vertex key is whatever string the caller declared.
- Genesis seeding, lineage-continuity scoring, and PPID verification (`_seed_genesis_from_parent_bg`, `_score_lineage_continuity_bg`, `parent_uuid=_parent_agent_id`) all inherit the unresolved handle.
- When a fork declares its parent by the bucket handle (e.g. `parent_agent_id="mcp_20260428"`), the lineage DAG anchors on a coarse `mcp_{date}` vertex that conflates every agent sharing that handle.

PR #700 (merged) appended a `uuid8` fragment to **future auto-generated** handles to narrow collisions. It did **not** change the storage key, demote the handle, or resolve `parent_agent_id` to a UUID — so the fork-into-handle pathway is still the design.

### This corrects A2/S4

`plan.md:268-289` (A2, 2026-04-21) inventoried label-as-identifier sites and concluded "internals operate on UUIDs," downgrading the concern to the lone S5 resident-fork locus. That audit scanned **resolution code, not DDL** — it did not cover the storage primary key, the lineage FK, or the AGE node key, all of which are handle-keyed. This proposal is the storage-layer half A2 did not reach.

## 2. Target state

| Tier | Today | Target |
|---|---|---|
| Identity key | `core.agents.id` = TEXT handle (PK) | `core.agents.id` (or `uuid`) = **UUID** (PK) |
| Public handle | same column as the PK | demoted to a **non-key, nullable** column (`public_handle` / `structured_id`) for display/grouping/provenance only |
| Lineage | `parent_agent_id` FK → handle | `parent_agent_id` FK → **UUID**, canonicalized at the onboard boundary; ambiguous bare-handle declarations rejected or flagged |
| AGE graph | `MERGE (a:Agent {id: <handle>})` | Agent nodes + SPAWNED/AUTHORED edges keyed on **UUID** |

`structured_id` keeps its *intended* job — a human-readable label for dashboards, KG provenance stamping, and model/date bucketing — and loses its accidental job as the storage key.

## 4. Phased migration (expand → backfill → dual-read → contract)

Each phase is its own migration slot (049+), its own draft PR, operator-gated. No phase drops or re-points the PK until Phase 4.

- **Phase 0 — decision (this doc).** Inventory + sequencing decision. Safe, docs-only.
- **Phase 1 — EXPAND (additive).** Add the UUID key column where the PK isn't already the UUID, plus a `public_handle` column; add nullable UUID columns to dependent tables; backfill from existing registry UUIDs. No PK change. **Mirror every column add in the Wave 3 shadow tables** so `schema_drift_check.sh` passes.
- **Phase 2 — DUAL-READ / WRITE.** Resolution layer resolves by UUID first, handle as fallback. `parent_agent_id` canonicalized to UUID at the onboard boundary (reject/flag ambiguous bare handles). New writes populate UUID keys; AGE writes UUID nodes (dual-writing a handle alias during the window).
- **Phase 3 — BACKFILL lineage.** Rewrite `parent_agent_id` handle→UUID across `core.identities` and AGE edges; split/de-dup conflated bucket vertices **while in-memory metadata still maps handle→uuid** (this mapping is the only thing that makes the split deterministic — see risk 1).
- **Phase 4 — CONTRACT (after the Wave 3 window).** Swap the PK to UUID, re-point/drop FKs, demote the handle to a non-key column. Final paired shadow-table update.

## 5. Sequencing & coordination

- **Wave 3 is the gating constraint.** The shadow replicas (`043`/`044`) cover exactly `core.agents` + `core.identities`, and Wave 3 proper is **operator-deferred until the §14 window closes (~2026-06-24)** (PR #706). The CONTRACT phase must land **after** that window; the EXPAND phases must update the shadow tables in lockstep (`schema_drift_check.sh`).
- **Single-writer surfaces.** Migration slots *and* identity/onboarding are both writer-locked (CLAUDE.md). This doc is the coordination artifact; re-check open PRs on `cirwel/unitares` **and** `cirwel/unitares-governance-plugin` before each phase.
- **Recommendation:** land Phase 0 (this proposal) now; begin Phase 1 EXPAND only once shadow-pairing is confirmed; hold Phase 4 until after ~2026-06-24.

## 6. Top risks

1. **AGE lineage re-key is not deterministically reversible after the fact.** A conflated `mcp_{date}` vertex (multiple agents on one node) can only be split correctly while the handle→UUID mapping still exists in memory/registry. Phase 3 must run inside that window, not later.
2. **Wave 3 shadow-drift gate.** Every expand step doubles (canonical + shadow); mis-sequencing breaks CI or the hourly divergence check.
3. **External callers still sending `agent_id`-as-handle** (continuity-token clients, the gov-plugin). The dual-read window must outlast client migration; CONTRACT cannot precede it.

## 7. Relation to existing work

- **Extends/corrects S4 (A2)** — covers the storage-layer gap the resolution-code audit missed.
- **Resolves the S5 resident-fork inversion locus** (`persistence.py:463`) for free once the handle is non-key (collision on a cosmetic label stops being identity-relevant).
- **Complements PR #700** — that made handles unique-ish; this makes handle uniqueness *irrelevant to identity*, which is the real fix.

## Amendment 2026-06-30 — simplification-council synthesis (no scope change)

The 2026-06-30 identity-layer **simplification council** (verdict: the identity
layer is *already simplified where it counts*; the implementation complexity is
incident-earned) ranked this migration the **single architectural lever with
real leverage** — and sharpened *why*. No change to §1–§7; this records the
decision-relevant framing.

### A. Root cause confirmed — the handle-PK is what keeps generating resolver band-aids

The council's finding is that the resolver-layer special-cases a simplifier
keeps wanting to cut are all **symptoms of the handle-as-PK**, not independent
debt. They dissolve into UUID equality once the handle is demoted:

- The **O(n) alias scan** at `src/mcp_handlers/support/agent_auth.py:409-421` —
  a linear scan over `agent_metadata.items()` matching the passed id against
  `label`/`public_agent_id`/`structured_id`. It exists *only* because a
  non-UUID handle cannot be a dict-key lookup; it collapses to the O(1)
  `if is_uuid:` branch directly above it (`:399-407`) once callers pass UUIDs
  and handles canonicalize at the boundary.
- The **"model+date OR UUID" dual-lookup** (`knowledge/handlers.py:479-495`, §3).
- The **self-ref alias rewrite** and the **S5 resident-fork inversion** (§7).
- **Lineage-DAG conflation** corrupting R1/R2 scoring inputs (§1).

**Strategic guidance (council):** *stop patching the resolver layer the
handle-PK keeps generating.* When a new resolver special-case is tempting,
prefer advancing this migration over adding another band-aid.

### B. Decision framing — this **lowers** urgency, it does not raise it

- **Blast radius is near-zero *today* for write-accountability.** The write gate
  already resolves to UUIDs internally (`require_agent_id` canonicalizes;
  context binding is UUID-keyed). The handle-PK is therefore mostly a
  *storage/display/lineage-graph* artifact, not a live write-decision hazard —
  so the cost of *waiting* is low. It generates resolver overhead and DAG noise,
  not wrong accountability verdicts.
- **BEAM / `db_connection` may re-key much of this for free.** The Wave-3
  substrate migration touches the same storage/connection layer; starting the
  UUID re-key *ahead* of BEAM risks doing work BEAM would redo. Where they
  overlap, sequencing should let BEAM land first rather than race it.

### C. Gate status refresh (2026-06-30)

### D. Net recommendation (unchanged in direction, firmer in rationale)

Hold at **Phase 0** (this doc is landed). Do **not** begin Phase 1 EXPAND as a
race against BEAM. In the meantime, treat any new resolver special-case as a
signal to advance *this* migration instead. Let the BEAM Wave-3 outcome inform
whether Phase 1 EXPAND is even the right vehicle, or whether a BEAM-native
re-key subsumes it. The responsible next action is operator/council sequencing,
not migration code.
