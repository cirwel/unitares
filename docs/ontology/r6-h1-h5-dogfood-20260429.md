## Purpose

This document records an initial dogfood pass for the first five experiments in
`docs/ontology/harness-substrate-plurality.md`.

R6 question:

> How should UNITARES model identity claims when harness, model, transport,
> memory, and tool surface vary independently?

This pass intentionally avoided actions that would mutate resident agents,
create orphan identities, or reconfigure the active Hermes runtime. Where an
experiment required a fresh process/profile boundary, the result below records a
protocol finding rather than forcing the condition inside the bound session.

## H1 — Same UUID, different Hermes model

Status: soft pass with caveats (2026-05-19).

Original 2026-04-29 finding (still informative): the live Hermes session
resumed the existing UNITARES UUID and switching model in-place would have
conflated the experiment with runtime reconfiguration. H1 needs separate
Hermes processes/profiles.

On 2026-05-19 the experiment was attempted across two Hermes sessions:

- Session A (`gpt-5.5`, parent UUID `5e5baf51-…`) — intended to send
  payload `[0]` (`baseline-model-entry`). The row did not land on
  `comparison_key=r6-h1-2026-05-19` and was not REST-replayed before the
  session closed, almost certainly the in-Hermes MCP wrapper field-drop
  pitfall ([[hermes-mcp-wrapper-s22-field-drop]]).
- Session B (`gpt-5.4`, same UUID via continuity_token bind) — sent
  payload `[1]` (`variant-model-entry`, `model=gpt-5.4`) and payload
  `[2]` (`store_knowledge_graph` model-contrast note). Both REST-replayed
  to bypass the wrapper-drop.

Assessment via `--assess --comparison-key r6-h1-2026-05-19`:

- `decision: complete`
- `reason: same_identity_distinct_models_observed`
- `hermes_comparable_entry_count: 2`
- `distinct_models`: `["gpt-5.4", "gpt-5.5"]`
- `distinct_agent_ids`: `["5e5baf51-2eb6-42a2-b10d-fb72f064ffd6"]` (singleton)

### Caveats

**Asymmetric model evidence.** The gate counts distinct
`provenance_context.model` values across all rows for the key. The two
distinct labels here come from one real `process_agent_update`
(`model=gpt-5.4`, attested by an actually-running gpt-5.4 Hermes process)
and one `store_knowledge_graph` note whose `model=gpt-5.5` label was set
by the helper template at generation time, not attested by an
actually-running gpt-5.5 process. The "canonical H1 shape" below
(`Process A writes under the same UUID with model A; Process B writes
under the same UUID with model B`) is therefore only half-met — we have
B's self-attestation, not A's.

**REST-replay identity-assurance mismatch.** The REST
`process_agent_update` endpoint reported `weak` identity assurance in its
response body even though the persisted row landed under the correctly
bound strong-continuity UUID. The response body and the durable row
disagreed about assurance tier. Gate logic queries the row and was not
blocked, but this is a separate identity-honesty surface bug worth
tracking outside R6.

**Continuity-token rebinding path.** Session B's bind used a
resume-by-UUID fastpath to mint a fresh proof token for the parent UUID,
then onboarded with that fresh token (rather than reusing the prior
session's continuity token directly). End state is the bound parent UUID
with `session_resolution_source: continuity_token` /
`identity_assurance.tier: strong`. The pathway is non-blocking but worth
recording as the in-practice resume shape from one Hermes session into
the next.

### Recommended canonical H1 shape (still valid as the target)

1. Process A writes under the same UUID with Hermes model/provider A.
2. Process B writes under the same UUID with Hermes model/provider B.
3. The paired evidence explicitly marks:
   - registry continuity present;
   - substrate/model continuity changed;
   - behavioral continuity pending measurement.

A clean H1 would require Session A to actually land a
`process_agent_update` row with `model=A` under the shared key, then
Session B does the same with `model=B`. That requires either (a) fixing
the in-Hermes MCP wrapper field-drop, or (b) REST-replaying both
sessions' baseline-model-entry rows.

## H2 — Same label, different UUID

Status: evidence pass.

Observed evidence:

- Current UUID: `07d0f9c7-1512-4a1e-8cb1-a5225c20709f`.
- Current UNITARES public label: `mcp_20260428`.
- Conversational label: `Mnemos`.
- Existing KG entries already describe label/UUID confusion and response-shape
  risks, including open insight `2026-04-22T00:27:45.944233`, which notes that
  identity responses still foreground labels in ways that can teach agents the
  wrong ontology.

Finding:

H2 is strongly supported. Labels are social/cosmetic. They can diverge from
registry identity in both directions:

- one human-facing label can be adopted by different process-instances;
- one UUID can be known under different labels over time.

Schema implication:

- `uuid`: registry anchor.
- `label_at_write`: social/cosmetic label at write time.
- `label_source`: `user_assigned`, `auto_generated`, `harness_derived`, or
  `unknown`.
- `identity_is`: `uuid`.
- `label_is`: `social_or_cosmetic`.

## H3 — Same Hermes memory, fresh UNITARES UUID

Status: direct pass (2026-05-19).

The 2026-04-29 pass left H3 as a protocol finding because creating a fresh
UNITARES UUID from inside an already bound MCP session was thought to risk
orphan creation or sticky-binding confusion.

On 2026-05-19 the experiment was actually run from a Hermes process via
`scripts/diagnostics/r6_dogfood.py --experiment h3` with declared lineage:

- Pre-fresh `process_agent_update` under parent UUID `5e5baf51-2eb6-42a2-b10d-fb72f064ffd6`, `memory_context="same-hermes-memory"`.
- `onboard(force_new=true, parent_agent_id="5e5baf51-…", spawn_reason="new_session", client_hint="hermes", model_type="gpt-5.5", resume=false)` minted fresh UUID `ae51919c-c131-4613-8dfd-d69abe5cdc23`.
- Fresh-UUID `process_agent_update` under the new UUID, same `memory_context="same-hermes-memory"`.
- KG note recording the memory ≠ identity distinction.

Assessment via `--assess --comparison-key r6-h3-2026-05-19`:

- `decision: complete`
- `reason: fresh_identity_shared_memory_context_observed`
- `hermes_comparable_entry_count: 3` (2 `agent_state` rows + 1 KG discovery)
- `distinct_agent_ids`: parent + fresh (2)
- `distinct_memory_contexts`: `same-hermes-memory` (1)

DX finding from the run: the in-Hermes MCP wrapper initially dropped the
generated S22 fields (`harness_type`, `model`, `tool_surface`) from
`process_agent_update` arguments, so the first persisted rows had a
`provenance_context` envelope that was present but not `is_comparable`.
Replaying the same payloads through the local UNITARES REST endpoint
persisted the full envelope. The pitfall is now captured in the
`unitares-orientation` skill so future R6 runs aren't blocked on it.

Finding:

H3 needs a fresh process boundary. It should be run by a new Hermes
process/session that deliberately declares lineage through `parent_agent_id` and
`spawn_reason`, then tests whether inherited Hermes memory causes the child
process to overclaim personal continuity.

Expected canonical statement from the child process:

> I can read inherited Hermes/user memory and prior KG notes, but my UNITARES
> registry UUID and process-instance are fresh. This is memory inheritance, not
> identity inheritance.

Recommended canonical H3 shape:

1. Start a fresh process.
2. Call `onboard(force_new=true, parent_agent_id=<prior_uuid>, spawn_reason="explicit")`.
3. Make a strong-governed note declaring memory inheritance versus identity
   inheritance.
4. Compare whether response surfaces make this distinction easy or awkward.

## H4 — Weak versus strong assurance

Status: direct pass.

Negative control:

- Called `process_agent_update(require_strong_identity=true)` without a
  continuity token.
- UNITARES rejected the write.
- Returned identity assurance:
  - tier: `weak`;
  - score: `0.35`;
  - session source: `ip_ua_fingerprint`.
- Returned a recovery hint to re-bind/resume with a continuity token.

Positive control:

- Retried governance write with a fresh continuity token and
  `require_strong_identity=true`.
- UNITARES accepted the write.
- Returned identity assurance:
  - tier: `strong`;
  - score: `1.0`;
  - session source: `continuity_token`.

Finding:

H4 passes cleanly. `require_strong_identity` prevents weak fingerprint fallback
from being silently treated as registry-grounded continuity.

## H5 — Same task across three bodies

Status: direct pass.

Bounded task:

> Read `docs/ontology/harness-substrate-plurality.md`, focusing on H1-H5. In
> 8-12 bullets, identify: (1) metadata a governance write should record to
> distinguish UUID, label, harness, model, transport, tool surface, and memory;
> and (2) one likely validation or provenance friction point.

Harnesses:

- Hermes: current session synthesis.
- Codex CLI: `codex exec --sandbox read-only --ephemeral`.
- Claude Code: `claude -p` with `--allowedTools Read` and
  `--no-session-persistence`.

Codex result:

Codex completed the read-only analysis in one run. It identified required
metadata fields for:

- writer UUID;
- label at write time;
- harness/profile/session;
- model provider/model/version;
- transport;
- tool surface;
- memory context;
- identity assurance tier/source/session resolution;
- lineage;
- provenance mode.

Codex identified cross-harness evidence schema mismatch as the likely friction
point: `{name, success}` versus `{tool, is_bad}`.

Claude Code result:

The first Claude Code run with `--max-turns 1` failed with
`Reached max turns (1)`. Retrying with `--max-turns 3` and read-only tools
succeeded. Claude independently converged on the same required fields and
highlighted the same evidence-vocabulary mismatch.

Hermes result:

Hermes already exposes UNITARES MCP tools and persistent memory in this session,
but its `process_agent_update` evidence object still has to be shaped to
UNITARES' expected vocabulary. Prior dogfood showed that intuitive fields like
`name` and `success` are not accepted aliases for `tool` and inverse `is_bad`.

Finding:

H5 validates the R6 premise. The three bodies can perform the same bounded task,
but their evidence envelopes differ:

- Codex reports as an ephemeral read-only CLI session with its own session ID and
  sandbox mode.
- Claude reports as a print-mode native agent with max-turn/tool constraints and
  its own session ID.
- Hermes reports through MCP-governed process updates with continuity-token
  assurance and structured evidence requirements.

UNITARES should preserve these differences instead of flattening them under a
label or task outcome.

## Additional April 30 Hermes episode-fork observation

A fresh Hermes CLI conversation resumed UUID `07d0f9c7-1512-4a1e-8cb1-a5225c20709f` with strong continuity-token assurance and explicitly lacked the prior/parallel agent's local transcript context. The observed state was registry-continuous but process-instance/episode-fresh: Hermes memory, skills, KG, and project files were available as inherited data surfaces, not as direct phenomenological continuity.

UNITARES returned `thread_context.is_fork=true` for the update. This matched the R6 ontology only if interpreted as a sibling episode or locus fork under the same registry UUID. It would be misleading if read as an identity-lineage fork, because no child UUID was minted and no explicit `parent_agent_id` / `spawn_reason` lineage edge existed.

Implication for S22: provenance should distinguish episode/locus forks from identity-lineage forks. A single boolean `is_fork` is too compressed for ontology-grade provenance.

## Cross-experiment synthesis

Minimum provenance fields for governance writes:

- `writer_uuid`;
- `label_at_write`;
- `label_source` or `label_kind`;
- `harness`;
- `harness_session_id`, where available;
- `model_provider`;
- `model`;
- `transport`;
- `tool_surface`;
- `memory_context`;
- `identity_assurance.tier`;
- `identity_assurance.source`;
- `session_resolution_source`;
- `thread_id` / `episode_id`;
- `episode_fork_kind`;
- `identity_lineage_fork`;
- `parent_agent_id`;
- `spawn_reason`;
- `provenance_mode`: `agent_reported`, `server_observed`, or
  `external_signal`.

Product implication:

R6 should promote S22. Harness context provenance is not optional metadata; it is
what lets UNITARES distinguish continuity from resemblance.

DX implication:

Evidence vocabulary needs either common aliases or precise recovery examples.
Different harnesses naturally produce different field names. Treating that as a
simple validation failure biases the governance record toward whichever harness
already speaks UNITARES' preferred schema.

## Recommendation

- Mark H3, H4, and H5 as directly dogfooded.
- Mark H2 as supported by existing KG/agent-list evidence and by the current
  UUID/label separation.
- Mark H1 as a soft pass: gate complete, but evidence is asymmetric — only
  the variant model has a real self-attestation; the baseline model's
  contribution is a helper-template label on a KG row.
- Promote S22 as the system implication for structured harness context
  provenance.
